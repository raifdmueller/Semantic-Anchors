= EARS-Requirements
:categories: requirements-engineering
:roles: business-analyst, qa-engineer, technical-writer

[%collapsible]
====
*Vollständiger Name*: Easy Approach to Requirements Syntax (Einfacher Ansatz für Anforderungssyntax)

*Kernkonzepte*:

* *Ubiquitäre Anforderungen*: "Das <System> muss <Anforderung>"
* *Ereignisgesteuerte Anforderungen*: "WENN <Trigger>, dann muss das <System> <Anforderung>"
* *Unerwünschtes Verhalten*: "FALLS <Bedingung>, DANN muss das <System> <Anforderung>"
* *Zustandsgesteuerte Anforderungen*: "WÄHREND <Zustand>, muss das <System> <Anforderung>"
* *Optionale Features*: "WO <Feature enthalten ist>, muss das <System> <Anforderung>"
* *Strukturierte Syntax*: Konsistente Vorlagen für Klarheit
* *Testbarkeit*: Anforderungen so geschrieben, dass sie überprüfbar sind

*Schlüsselvertreter*: Alistair Mavin (Rolls-Royce)

*Wann zu verwenden*:

* Sicherheitskritische Systeme
* Regulierte Industrien (Luft- und Raumfahrt, Automobil, Medizin)
* Wenn Anforderungsrückverfolgbarkeit wesentlich ist
* Große, verteilte Teams
====
